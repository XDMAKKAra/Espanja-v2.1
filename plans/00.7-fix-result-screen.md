## The bug (with visual evidence)

See screenshot in `plans/assets/00.7-current-result-screen.png` (marcel took this 2026-04-19). After a single 11-answer vocab exercise the result screen shows:

- Big headline: `ARVIOITU YO-ARVOSANA`
- Large `L` letter badge (orange)
- `9 / 11 oikein · 82%`
- `2 min 58 s`
- `B → C` (level delta)
- `I A B C M E L` band with the current estimate underlined

**Why this is wrong:**

1. **It's dishonest.** YTL grade bands (I/A/B/C/M/E/L at 10/20/35/50/65/80 points on a 299-point scale) are computed across six exam sections — listening, reading, grammar/vocab, writing. One vocab drill cannot produce an L. Showing L after 11 questions is a lie the user can see through in 2 seconds.

2. **It's not pedagogical.** The user learned something in this exercise — which words, which grammar points went wrong. That is what a learning product should surface. A grade band is a verdict, not a lesson.

3. **It kills trust.** A user who sees a fake L prediction assumes the rest of the app's "adaptive" claims are equally fake. We cannot afford this perception a week before marketing work begins.

## The fix — two parts

### Part 1: Rebuild the exercise result screen

Remove the YO-grade prediction from the **per-exercise result screen entirely.** That screen should answer three questions, in order:

1. **How did this exercise go?** — raw facts, no interpretation.
2. **What did the user learn / still need to practice?** — per-item pedagogical feedback.
3. **What's next?** — a direct next step, ideally targeting the weakness shown in (2).

### Part 2: Gate the YO-arvosana prediction on the dashboard

Move the YO-arvosana prediction to **the dashboard only**, and gate it behind data thresholds so it never lies.

**Threshold ladder:**

| Harjoituksia tehty | Näytetään dashboardissa |
|---|---|
| < 10 | `Tee vähintään 10 harjoitusta 3 eri osa-alueesta, jotta voimme arvioida YO-tasoasi.` Ei arvosanakirjainta. |
| 10–29 | `Alustava taso: {grade}` — small text, muted color, no letter badge. Clickable → opens explainer. |
| 30–99 | `Arvioitu YO-taso: {grade}` — normal size, with confidence bar (e.g. `■■■□□ 3/5`). Clickable → explainer. |
| 100+ with coverage across all 4 sections | `Arvioitu YO-arvosana: {grade}` — full size, confidence bar, "päivitetty {date}" caption. Clickable → explainer. |

**"Coverage across all 4 sections"** means: at least 10 exercises each in vocab, grammar, reading, writing. If the user has done 200 vocab exercises but zero writing, they stay in the 30–99 tier even though total > 100. This is critical — the grade prediction is not allowed to be built on a single-section dataset.

**The explainer modal** (open on click of the grade widget): show a breakdown of which sections contributed, how many exercises in each, current per-section CEFR estimate, and a caveat sentence: `Tämä on arvio nykyisen harjoitteludatan perusteella, ei virallinen YTL-arvio.`

### Part 3: Where the grade is actually computed

Find the backend function that currently produces the L on the result screen. Likely candidates:
- `routes/exercises.js` (vocab grade endpoint)
- `routes/progress.js` (dashboard data)
- Possibly a `lib/grading.js` or inline computation in `app.js`

Audit the call site and:

1. **Remove the per-exercise grade call.** The exercise result endpoint should return `{ correct, total, timeMs, items: [{ prompt, userAnswer, correct, concept }] }` — no grade, no level delta.
2. **Move the grade computation to the dashboard endpoint** (likely `/api/progress` or similar). That endpoint returns `{ exerciseCount, coverage: { vocab, grammar, reading, writing }, gradeEstimate: 'L'|'E'|...|null, confidence: 0-5 }`. Return `null` for gradeEstimate when thresholds aren't met — the frontend decides how to render null vs. a grade.
3. **Unit test the threshold logic.** `tests/grade-threshold.test.js` — given counts {vocab:5, grammar:5, reading:0, writing:0}, expect `gradeEstimate === null`. Given {vocab:50, grammar:50, reading:50, writing:50}, expect a non-null grade with confidence ≥ 3.

## What NOT to do

- Don't keep the YO-arvosana on the result screen "just smaller" or "as a preview". Remove it entirely. The compromise is worse than either extreme.
- Don't compute the grade client-side. The backend owns truth.
- Don't invent a new "exam readiness score" to replace the grade. Raw score + concept feedback is enough for this screen.
- Don't touch the grading algorithm itself in this pass — just move where it's called and add the thresholds. Algorithm hardening is Pass 2.5's job.
- Don't remove grade from other legitimate places (e.g. the dedicated exam-simulator completion screen, which has enough data to justify one).

### Bug 1 — Onboarding + placement replay every login

**User report:** "I've logged in with the same credentials many times and it always asks me to do the first questions how long i've studied spanish etc."

**Two suspected flows replaying:**

1. **Onboarding** gates on `user_profile.onboarding_completed` at `js/screens/onboarding.js:59`. The POST to `/api/profile` at `onboarding.js:450-454` is wrapped in `try { ... } catch {}` at lines 469-471 that **silently swallows errors**. If the save fails (RLS denial, schema drift, CORS, 500), the flag never persists and onboarding replays forever.
2. **Placement** gates on a row existing in `diagnostic_results` (`routes/placement.js:200-225`). If the user bails mid-placement, or the submit endpoint fails, no row → always shows as incomplete.

**Fix:**
- **Remove silent catches.** In `js/screens/onboarding.js` `saveAndFinish` + `skipOnboarding`, log the error to console AND surface it to the user with a toast ("Tallennus epäonnistui, yritä uudelleen"). Never silently proceed — that's what caused the bug.
- **Reproduce with a test user.** Create a fresh account, complete onboarding, log out, log back in. If onboarding replays, inspect Network tab for the `/api/profile` POST status. Fix root cause (likely RLS policy or a field validation returning 400).
- **Add server-side telemetry.** In `routes/profile.js`, log every failed profile save with user id + error code + attempted fields, so future regressions surface in logs not in screenshots from users.
- **Add a regression test.** `tests/onboarding-persistence.test.js` — POST `/api/profile` with `{ onboarding_completed: true }`, then GET `/api/profile`, assert `onboarding_completed === true`. Same for placement: POST `/api/placement/submit`, then GET `/api/placement/status`, assert `completed === true`.

### Bug 2 — No login entry point on landing page

**User report:** "There is no log in page only 'aloita ilmaiseksi' on the right."

**Root cause:** `index.html:762-768` nav only has one CTA — "Aloita ilmaiseksi" pointing to `app.html`. Returning users have to click the signup CTA, then find the tab inside `app.html` to toggle to login. `app.html` does contain both forms, but from the landing page there's no affordance that says "I already have an account."

**Fix:**
- Add a second nav CTA: `<a href="app.html?mode=login" class="nav-link-login">Kirjaudu sisään</a>` directly before the existing `<a href="app.html" class="nav-cta">Aloita ilmaiseksi</a>` on `index.html:768`. Style it as a text link, not a button — subordinate to the primary signup CTA.
- Mobile: ensure the login link is visible in the mobile nav layout too (currently the nav-links div at 762-767 is desktop-only; the login link needs to be reachable on narrow widths).
- In `app.html` / `js/screens/auth.js`: read `?mode=login` from URL params on boot and preselect the login tab instead of the default register tab. One-line addition at the top of `initAuth`.
- Apply to every page that has the landing nav: `index.html`, `pricing.html`, `blog/index.html`, `diagnose.html`, and every individual blog post under `blog/`. Grep for `nav-cta">Aloita ilmaiseksi` to find them all.

### Bug 4 — Manual taso-valitsin overrides per-topic mastery

**User report:** "minun nettisivuni pitäisi tietää minun tasoni eri kohdissa esim sanasto kirjoittaminen yms, joten tuo että voin valita tasoni tuolla alhaalla on aika huono."

**Root cause:** Vocab, grammar, and reading mode pages each render a manual B/C/M/E/L (and I/A on vocab) "TASO" picker that the user must click before starting. The picker overrides whatever the system already knows about the user from `user_mastery` and `diagnostic_results`. This defeats the entire adaptive system: a user who placed at C in ser_estar but B in subjunctive sees the same default-active button on every screen and either guesses, picks their target grade (too hard), or sticks with the default (B for vocab, C for grammar/reading) regardless of their real level.

**Real-world failure mode:** B1 student who tested at M in vocabulary clicks the default-active "B" button without thinking, gets too-easy items, gets bored, leaves. Or: same student clicks "M" because that's their target grade, gets crushed, leaves.

**Code locations to remove (all in `app.html` unless noted):**

1. `app.html:657-665` — `#vocab-page-level-picker` (vocab mode page)
2. `app.html:700-706` — `#grammar-page-level-picker` (grammar mode page)
3. `app.html:856-862` — `#reading-page-level-picker` (reading mode page)
4. `app.html:966-974` — legacy `#level-picker` (vocab inline)
5. `app.html:1034-1040` — legacy `#grammar-level-picker` (grammar inline)
6. `app.html:1059-1065` — legacy `#reading-level-picker` (reading inline)

**⚠️ DO NOT remove `app.html:741-744` — `#verbsprint-duration-picker`.** It uses the same `.level-picker` CSS class but represents *duration* (10/20 seconds), not CEFR level. It's a real choice the user should make. Verify by checking `data-duration` vs `data-level` attribute.

**Listeners to remove or rewrite:**

- `js/main.js:132-178` — page-level-picker listeners + state.level / state.grammarLevel / state.readingLevel reads. Replace with mastery lookup (see fix below).
- `js/main.js:329-352` — legacy `#level-picker`, `#grammar-level-picker`, `#reading-level-picker` listeners. Delete entirely; the elements are gone.
- `js/screens/dashboard.js:629-654` — picker active-class toggling on dashboard navigation. Delete; pickers don't exist anymore.
- `app.js:844-880` and `app.js:1070-1180` — duplicate of the above in the legacy bundled `app.js`. The codebase is mid-migration from `app.js` to `js/screens/*.js`; remove from both. Confirm with marcel which one is actually loaded — if `app.js` is dead code, mark for deletion in a follow-up commit but at minimum don't leave conflicting handlers.

**Fix (the new behaviour):**

When user clicks "Aloita harjoittelu" on vocab/grammar/reading mode pages, the client must:

1. Read the selected topic from the topic-picker (which stays — topic choice is real choice, level is not).
2. Call new endpoint `GET /api/mastery?topic={topic}` (add to `routes/progress.js` if it doesn't exist — verify first; `user_mastery` table is queried in `routes/exercises.js:895` already, so the data is there).
3. If a mastery row exists for this topic, use `mastery.current_level` for the exercise request.
4. If no mastery row exists (new topic for this user), fall back to the user's global placement level from `diagnostic_results.placement_level`.
5. If no placement either (placement test not taken yet), fall back to `"B"` for vocab and `"C"` for grammar/reading (current defaults — but this case should be rare since onboarding now forces placement).

State variables `state.level`, `state.grammarLevel`, `state.readingLevel` become derived-on-demand instead of picker-driven. They can stay as variables but their values come from the mastery lookup, not from a DOM read.

**"Haasta itseäsi" affordance (deferred to Pass 1, do NOT add this pass):**

Some users will legitimately want to try a harder level. After Pass 1's design system ships and the side-panel pattern exists, a small "Kokeile vaikeampaa →" link can sit in the corner of the exercise page (not on the mode page, not in the start flow) for users who explicitly want to push themselves. This pass: just remove the picker. Pass 1: add the optional escape hatch.

**Regression test:**

`tests/no-manual-level-picker.test.js` — start a session for a user with `user_mastery` row `{ topic: "ser_estar", current_level: "C" }`. Trigger vocab generation request. Assert the request body has `level: "C"`, not `"B"`. Repeat for grammar topic with mastery `"M"` — assert `"M"`, not the default `"C"`.

**UI note for the empty-space:**

Removing 6 picker rows leaves a visual gap above the "Aloita harjoittelu" button. Don't paper over it — let the topic-picker breathe. Pass 1 will rework these mode pages anyway. If the gap looks really bad in the meantime, replace it with a small "Tasosi: M" status line that reads from mastery (read-only, no buttons).

### Bug 3 — Narrow desktop layout (deferred to Pass 1)

**User report:** "This website looks like it's made only for phone use. It's very narrow."

**Root cause:** `style.css` is mobile-first. Every screen container caps at 360–680px (lines 391, 642, 1053, 1104, 1339, 1543, 2052, 2344). On a 1920px monitor the content uses ~640px and leaves ~1280px of empty margin. The sidebar is fine (220px, `style.css:146-151`); the content area is the problem.

**Deferred to Pass 1 (design system)**, because fixing it properly requires layout tokens, responsive breakpoints, and per-screen thinking. A quick "widen the container" fix will break alignment everywhere.

**But do this now, this pass:** add a `plans/01-design-system-step1.md` explicit acceptance criterion: "Desktop layout (≥1024px) must use ≥960px of content width on dashboard and every exercise screen. Empty desktop margins are a bug, not a feature."

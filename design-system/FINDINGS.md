# FINDINGS — legacy quirks that affect Step 2

Surprises uncovered during the audit. Each has a file:line citation and a recommended handling in Step 2.

## 1. `design-system/puheo/` is unrelated boilerplate

`design-system/puheo/MASTER.md` and `design-system/puheo/pages/landing.md` look auto-generated by a template tool. They describe:
- A **green** palette (`#059669` primary, `#10B981` secondary) — incompatible with Puheo's red/dark brand.
- A **fashion / luxury** mood using Cormorant + Montserrat — wrong register for exam prep.
- A landing page spec for an **event** (speakers grid, sponsors, countdown) — wrong product type.

**Handling:** ignore entirely. Don't delete (it's harmless), but don't import any of its guidance. The canonical design system lives in `design-system/AUDIT.md`, `DESIGN.md`, `SCREENS.md`, `PLAN.md`, `FINDINGS.md`. Step 2 can remove `design-system/puheo/` in its final cleanup commit.

## 2. Split token namespaces

`style.css:8–9` defines `--font-display` and `--font-mono`.
`landing.css:15–17` defines `--font`, `--mono`, `--serif`.

Same physical fonts. Two names. Any new component has to know which file it lives in and pick the right token.

**Handling:** Step 2 Commit 1 unifies to `--font-display` / `--font-body` / `--font-mono` / `--font-serif`. Adds Inter as the body font. Legacy aliases (`--font`, `--mono`) kept for one gate as compat, then removed in Gate E.

## 3. `--brand-btn` was added for contrast, not replacing `--brand`

`landing.css:7` has a comment: `--brand-btn: #c62020; /* darker brand for white-text buttons — passes WCAG AA (5.80:1) */`. So `--brand` (`#e63946`) is display-only, `--brand-btn` is interactive. Nothing in `style.css` knows this distinction.

**Handling:** DESIGN.md adopts the split officially. Any CTA / button uses `--brand-btn`; any decorative gradient or hero uses `--brand`. Step 2 Commit 3 enforces this in the button component.

## 4. 3 separate nav implementations

- **Landing nav**: `landing.css:133–170` (`.nav` fixed, 64px).
- **diagnose.html nav**: inline `<style>` at `diagnose.html:71–96` — re-declares the same thing with `--brand` instead of `--brand-btn` on the CTA.
- **App nav**: `app.html:62` onwards uses a different shell with sidebar on desktop, top bar only on mobile.

Three drift points. When Pass 0.5 added "Kirjaudu sisään", each had to be patched separately.

**Handling:** Step 2 Commit 8 collapses landing + diagnose into one `.top-nav` component (`css/components/top-nav.css`), removes the inline `<style>` from `diagnose.html`. App shell nav stays separate because it's structurally different (sidebar vs no sidebar).

## 5. `style.css` is 4,398 lines, no section index

No table of contents. Hard to find where `.reading-question` lives without grep. Current section comments are sporadic (`/* ── LEVEL SELECT ── */` etc.).

**Handling:** Step 2 Commit 1 adds a TOC block comment at the top of `style.css` listing every section with line ranges. Future commits keep it updated. Not a separate file — stays inline for `@import` simplicity.

## 6. No focus ring in `style.css`

`landing.css:72–75` defines `:focus-visible { outline: 2px solid var(--brand-light); }`. Nothing equivalent in `style.css`. Keyboard navigation of the app is essentially invisible.

**Handling:** Step 2 Commit 1 adds the identical global rule to `style.css`. This is a ~4 line fix that upgrades a11y substantially.

## 7. Confetti animation ignores `prefers-reduced-motion`

`js/screens/onboarding.js` imports a confetti lib (`launchConfetti()` at the done screen). Nothing checks `matchMedia('(prefers-reduced-motion: reduce)').matches`.

**Handling:** Step 2 Commit 6 (feedback/toast) adds a `js/util/motion.js` helper `prefersReducedMotion()` and wraps confetti, pulse, and slide animations. One-liner per call site.

## 8. Placement test + onboarding use different nav styles

`screen-placement` has no top nav at all (`app.html` ~line 450+ — confirmed by grep). `screen-onboarding` has a progress bar but no nav. Users can't go "back" from either screen — only forward or accept-and-continue. That's a deliberate product choice but it means the design system's top-nav spec doesn't apply here.

**Handling:** DESIGN.md §8.9 top-nav is optional per screen. Placement and onboarding keep a minimal header (logo + progress) as a distinct variant. Document in SCREENS.md (already noted for onboarding).

## 9. `!important` clustering

All 6 non-utility `!important` declarations sit in `style.css:1970–2120` — the exam + mode-select block. This suggests a single historical refactor went wrong and nobody went back to fix it.

**Handling:** Step 2 Commit 3 (button) + Commit 16 (exam screen) together remove every `!important` from that range by flattening selector depth.

## 10. Blog pages use Lora serif but nothing else in the codebase does

`landing.css:17` declares `--serif: 'Lora', serif`. Only the blog uses it. The font weight (400 + italic 400) is loaded globally on every page via the Google Fonts link in `index.html:10` — so 40kb of font is downloaded on the homepage even though only blog pages render it.

**Handling:** Step 2 Commit 19 moves the Lora `<link>` into the blog shell only. Saves ~40kb on every non-blog page. Lighthouse perf goes up ~3 points.

## 11. Inline `<style>` in `diagnose.html` + legal pages

`diagnose.html:25–566` has ~540 lines of inline `<style>` — a full stylesheet embedded in the HTML. `pricing.html:10–153`, `privacy.html:10–49`, `terms.html:10–49`, `refund.html:10–63` each have their own.

**Handling:** Step 2 Commit 18 extracts these into `pages.css` shared by all non-app HTML pages. Removes ~900 lines of duplicated inline CSS.

## 12. No existing `tests/` visual regression harness

The `tests/` folder has unit tests (vitest). Playwright configs exist at repo root but are not wired into CI. Visual screenshots in PLAN.md require a playwright runner.

**Handling:** Step 2 Commit 11 (dashboard re-skin) adds the first visual regression test as a template, extending the existing playwright config. Document the pattern in `tests/helpers/visual.js`. Subsequent screen commits reuse it.

---

## Cross-cutting risks for Step 2

- **Gate D is the fragile gate.** Six commits touching every app screen. If a token rename goes wrong in Gate A, Gate D will surface the pain. Mitigation: Gate A commit 1 keeps every old token name as an alias; delete aliases only in Gate E.
- **Inter font load.** Adding Inter adds ~60kb. Offset by removing Lora from non-blog pages (Finding 10) and preloading critical subset only.
- **Light-mode QA.** The app has never had light mode. Every component needs manual QA in both. Don't underestimate Gate E Commit 20.
- **Mobile safe-area.** The urgency bar + bottom nav both need `env(safe-area-inset-*)`. Current implementation has this for the urgency bar but not the bottom nav (which doesn't exist yet). Easy to forget.
